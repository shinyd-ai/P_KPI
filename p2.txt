# Layout Audit Skill

## 목적

이 Skill은 responsive-analyzer가 모바일 레이아웃 문제를 빠뜨리지 않고 찾아내도록 체크 항목을 제공한다.

## 절차

1. **구조 파악**: `app/layout.tsx` 를 읽어 전체 레이아웃 골격 확인
2. **사이드바 확인**: `components/layout/Sidebar.tsx` 에 모바일 분기(`md:hidden` 등) 있는지 확인
3. **고정 너비 탐색**: Grep으로 `w-\d`, `min-w-`, `max-w-` 패턴 검색
4. **flex/grid 확인**: 반응형 prefix 없이 쓰인 `flex`, `grid-cols-` 탐색
5. **버튼/터치 확인**: `py-1`, `p-1` 처럼 터치 영역이 너무 작은 버튼 탐색
6. **오버플로 확인**: `overflow-hidden` 없이 가로로 넘칠 수 있는 구조 탐색

## 검색 패턴 (Grep 기준)

```
고정 너비:    w-\d+(?!\/), min-w-\d
반응형 미적용: ^(?!.*md:).*flex-row
터치 영역:    py-1\b, p-1\b
```

## 산출물 형식

문제를 심각도 3단계로 분류한다.

- **높음**: 모바일에서 레이아웃이 바로 무너지는 것
- **중간**: 좁은 화면에서 어색하거나 스크롤이 생기는 것
- **낮음**: 사용은 가능하지만 개선하면 더 좋은 것

- 분석 보고서는 설계자가 바로 설계에 사용할 수 있어야 한다

# Mobile-First Design Skill

## 목적

이 Skill은 responsive-designer가 분석 결과를 구체적인 Tailwind 변경 계획으로 바꾸도록 돕는다.

## 원칙

- **모바일 퍼스트**: 기본 클래스는 모바일, 큰 화면은 `md:` prefix로 덧붙인다
- **네비게이션 전환**: 사이드바(PC) ↔ 하단 탭 바(모바일)
- **콘텐츠 너비**: 모바일은 full-width, PC는 max-w 유지
- **터치 영역**: 최소 `py-3 px-4` 이상 권장

## P_KPI2 앱 기준 표준 패턴

| 상황 | 모바일 클래스 | PC 추가 클래스 |
|------|--------------|---------------|
| 사이드바 | `hidden` | `md:flex` |
| 하단 탭 바 | `fixed bottom-0` | `md:hidden` |
| 레이아웃 방향 | `flex-col` | `md:flex-row` |
| 콘텐츠 패딩 | `px-4 py-4` | `md:px-6 md:py-6` |
| 하단 여백 (탭 바 공간) | `pb-20` | `md:pb-0` |
| 그리드 | `grid-cols-1` | `md:grid-cols-2` |

## 절차

1. 분석 보고서의 높음 항목부터 처리 방안을 설계한다
2. 각 항목에 대해 `현재 클래스 → 변경 클래스` 형태로 명시한다
3. 신규 파일 필요 시 파일명과 역할만 계획서에 적는다 (구현은 하지 않는다)
4. 변경 우선순위를 1~3단계로 나눠 구현자가 순서를 알 수 있게 한다

## 품질 기준

- 계획서만 보고도 구현자가 어떤 클래스를 어디에 쓸지 알 수 있어야 한다
- PC 기존 동작을 깨뜨리는 계획을 포함하지 않는다
- 계획에 없는 파일을 범위에 넣지 않는다

# Responsive Implement Skill

## 목적

이 Skill은 responsive-implementer가 계획서대로 코드를 수정할 때 실수를 줄이도록 돕는다.

## P_KPI2 프로젝트 주의사항

- Tailwind CSS 4 사용: `tailwind.config.ts` 없음, `globals.css` 에 `@import "tailwindcss"` 방식
- Next.js App Router: `"use client"` 필요한 컴포넌트 확인
- 기존 PC 레이아웃 (`md:` 이상)을 건드리지 않는 것이 원칙

## 절차

1. 계획서에서 변경 우선순위 1번 항목을 확인한다
2. 해당 파일을 Read로 읽는다
3. 변경할 클래스 또는 구조를 Edit으로 정확히 수정한다
4. 수정 후 파일을 다시 Read로 읽어 의도대로 됐는지 확인한다
5. 구현 로그에 완료로 기록하고 다음 항목으로 넘어간다

## 안전 수칙

- 한 번에 하나의 파일만 수정한다
- 기존 클래스를 제거할 때는 제거 이유를 로그에 남긴다
- `className` 문자열이 길어지면 줄바꿈으로 가독성을 유지한다
- 신규 파일 생성 시 기존 컴포넌트 스타일 패턴을 먼저 참고한다

## 하단 탭 바 표준 구조 (BottomNav)

```tsx
"use client";
import Link from "next/link";
import { usePathname } from "next/navigation";

const NAV_ITEMS = [
  { href: "/",        icon: "📊", label: "대시보드" },
  { href: "/goals",   icon: "🎯", label: "목표" },
  { href: "/monthly", icon: "📅", label: "월간" },
  { href: "/daily",   icon: "📋", label: "일간" },
];

export default function BottomNav() {
  const pathname = usePathname();
  return (
    <nav className="fixed bottom-0 left-0 right-0 z-50 bg-white border-t border-zinc-200 md:hidden">
      <div className="flex">
        {NAV_ITEMS.map((item) => (
          <Link
            key={item.href}
            href={item.href}
            className={`flex-1 flex flex-col items-center gap-1 py-3 text-xs transition-colors ${
              pathname === item.href ? "text-blue-600" : "text-zinc-400 hover:text-zinc-600"
            }`}
          >
            <span className="text-lg">{item.icon}</span>
            {item.label}
          </Link>
        ))}
      </div>
    </nav>
  );
}
```

## 품질 기준

- 수정 전후 Read로 확인한다
- 구현 로그에 변경 내용을 빠뜨리지 않는다
- 브라우저 실행 없이 "동작 확인됨"이라고 쓰지 않는다
